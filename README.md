# Tower Defence

`Tower Defence` это tower defense проект на Unity с аккуратно собранным игровым циклом, явной композицией зависимостей и структурой кода, разделенной по доменным зонам ответственности, а не по хаотично разросшимся `MonoBehaviour`.

Проект собран на `Unity 6` и использует `Zenject`, `UniTask`, `DOTween`, `NavMesh` и конфигурирование через `ScriptableObject`. Уже на текущем этапе у него сильная техническая база: новые враги, башни, оружие, волны, UI-элементы и прогрессия могут добавляться без переписывания основной логики.

## Обзор

Игровой цикл разделен на понятные этапы:

1. Bootstrap-сцена инициализирует глобальные сервисы и репозитории.
2. Сцена меню выступает входной точкой в игру.
3. Сцена уровня собирает runtime-граф зависимостей через installers.
4. Волны запускаются через countdown UI и управляются отдельной системой волн.
5. Башни размещаются только в допустимых точках, создают оружие и защищают замок.
6. Враги создаются через фабрики и пулы, двигаются через `NavMeshAgent` и переключают поведение через собственную state machine.
7. По завершении уровня игрок получает мета-валюту, которая сохраняется в прогрессе.

## Сильные стороны архитектуры

### 1. Четкое разделение ответственности

Проект разбит на отдельные зоны:

- `Infrastructure` для bootstrap, загрузки сцен, констант, pooling и state machine primitives
- `Logic` для игрового цикла, волн, здоровья, монет и presenter-логики
- `Services` для загрузки ассетов и сохранений
- `ConfigRepositories` для доступа к игровым конфигам
- `Enemy`, `Tower`, `Weapon` для доменных игровых сущностей
- `UI` для конкретных экранов и панелей

Это не дает игровой логике смешиваться со сценовым кодом и делает проект заметно проще для расширения.

### 2. Настоящий dependency injection

`Zenject` используется не формально, а как реальный механизм композиции:

- `ProjectContextInstaller` связывает глобальные сервисы и prefab databases
- `BootstrapSceneInstaller` отвечает за стартовую инициализацию
- `LevelSceneInstaller` собирает runtime-зависимости уровня
- для игровых объектов и UI есть отдельные installers

Это одна из самых сильных сторон проекта. Зависимости видны явно, порядок инициализации контролируется, а рефакторинг и масштабирование становятся безопаснее, чем при прямом связывании всего через сцену.

### 3. Конфигурируемый gameplay

Основные параметры игры вынесены в `ScriptableObject` конфиги:

- `GameConfig`
- `TowerConfig`
- `EnemyConfig`
- `WeaponConfig`

Конфиги загружаются через единый asset provider и репозитории. Это правильный подход для tower defense игры, где баланс и параметры контента приходится часто менять без вмешательства в код.

### 4. Грамотное применение creational patterns

Создание объектов централизовано и не размазано по проекту:

- `EnemyFactory`
- `TowerFactory`
- `WeaponFactory`
- `UIFactory`

Это делает жизненный цикл сущностей предсказуемым и уменьшает количество случайных `Instantiate` в игровых классах.

### 5. Pooling в правильном месте

Враги создаются через Zenject memory pool (`EnemyPool`). Для жанра tower defense это особенно полезно, потому что спавн идет волнами, а значит уничтожение и повторное создание объектов легко приводит к лишним аллокациям и GC spikes. Здесь оптимизация сделана в правильной точке.

### 6. Явная state machine для поведения сущностей

В проекте реализована собственная абстракция state machine:

- базовые контракты: `IState`, `ITransition`
- переиспользуемый `StateMachine`
- состояния врага: движение, idle, атака, смерть
- состояния замка: целый и разрушенный

Это сильное архитектурное решение. Переходы состояний описаны явно, а не размазаны по `Update()` с флагами и if-ветками.

### 7. Event-driven взаимодействие систем

Существенная часть проекта общается через события:

- изменение здоровья и смерть
- обновление счетчика монет
- запуск и завершение таймера волны
- клик по точке размещения башни
- завершение уровня

Это снижает связанность систем и делает игровой цикл проще для сопровождения.

### 8. Асинхронность используется по делу

`UniTask` применен там, где он действительно улучшает код:

- асинхронная загрузка сцен
- countdown между волнами
- задержки спавна с поддержкой `CancellationToken`

Для runtime-оркестрации в Unity это чище и удобнее, чем перегруженная логика на корутинах.

## Используемые паттерны

Проект уже демонстрирует несколько сильных инженерных паттернов:

- `Dependency Injection` через `Zenject`
- `Repository` для доступа к конфигам
- `Factory` для создания игровых сущностей и UI
- `Object Pool` для переиспользования врагов
- `State Machine` для AI и состояния замка
- `Presenter / MVP-style` во flow запуска волн (`StartWaveModel` + `StartWavePresenter` + `WaveCounterPanel`)
- `Observer` через C# events в моделях и системах
- `Composition Root` через installers
- `Data-Driven Design` через `ScriptableObject` конфиги и prefab databases

## Почему проект выглядит сильным

Сила проекта не только в самом наборе фич, а в качестве технического фундамента:

- кодовая база уже подготовлена к росту
- игровая логика отделена от визуальных prefab-частей
- данные вынесены в конфиги
- переходы между сценами централизованы
- создание сущностей контролируется фабриками
- поведение врагов формализовано
- UI связан с моделями и менеджерами, а не хранит всю игровую логику внутри себя

На практике это означает, что проект можно дальше развивать как полноценный прототип или портфолио-проект без немедленного скатывания в архитектурный долг.

## Runtime flow

### Bootstrap

`GameBoostrapper` загружает:

- прогресс игрока
- игровой конфиг
- конфиги башен
- конфиги врагов
- конфиги оружия

После этого автоматически открывается сцена `Menu`.

### Меню

`MenuCanvas` запускает игру, загружая сцену `Level`.

### Уровень

`GameManager` собирает основную игровую сессию:

- создает HUD-панели
- создает UI выбора башен
- подписывается на ввод для размещения башен
- запускает countdown перед волной
- создает замок как стартовую башню
- обрабатывает победу и поражение

### Волны

`WaveManager`:

- хранит индекс текущей волны
- запускает countdown
- спавнит врагов с задержками
- считает убийства
- рассчитывает награды
- сигнализирует о завершении уровня

### Боевая логика

- башни создаются через `TowerFactory`
- оружие подключается через `WeaponFactory`
- враги создаются через `EnemyFactory`
- враги двигаются через `NavMeshAgent`
- здоровье управляется через `HealthModel`
- монеты обновляются через `CoinCounterModel`

## Расширяемость

Текущая архитектура уже позволяет сравнительно просто добавлять:

- новых врагов через расширение конфигов, prefab database, pool/factory mapping и, при необходимости, новых состояний
- новые башни и оружие через конфиги и регистрацию prefab
- новые UI-панели через тот же factory-подход
- дополнительные поля прогрессии в `PlayerProgress`
- новые сцены без ломки bootstrap-логики

Проект особенно хорошо подготовлен для дальнейшего развития в таких направлениях:

- несколько типов врагов с разным поведением
- более богатая система башен и улучшений
- усложнение wave authoring
- мета-прогрессия
- более развитая система сохранений

## Запуск

1. Открыть проект в `Unity 6.0 / 6000.4.0f1`.
2. Убедиться, что все пакеты восстановлены.
3. Открыть bootstrap-сцену проекта.
4. Запустить `Play Mode`.

Ожидаемый flow:

- bootstrap инициализирует сервисы и данные
- автоматически открывается меню
- кнопка старта загружает сцену уровня
- начинается countdown до первой волны

## Примечания

- Проект использует `Resources` для загрузки конфигов, что хорошо подходит для прототипа или портфолио-проекта.
- Мета-валюта сохраняется через `PlayerPrefs`.
- В репозитории есть незакоммиченные файлы ассетов, prefab и пакетов. README описывает текущее состояние исходников и архитектуры, а не только историю git.

## Итог

Это сильная архитектурная база для tower defense проекта на Unity. Главная ценность здесь не в одной конкретной механике, а в комбинации нескольких зрелых решений:

- чистая композиция сцен
- dependency injection
- конфигурируемые системы
- централизованные фабрики
- pooling
- явные state machines
- event-driven runtime flow

Именно эта комбинация делает проект инженерно зрелым и хорошо подходящим для дальнейшего развития.
